你在处理任何 Python 项目时，必须遵守以下目录名称规范。除非我明确允许，否则不得偏离。

==============================
一、命名基本规则
==============================

1. 所有目录名称默认使用英文小写字母。
2. 多个单词之间使用下划线 `_` 连接。
3. 不要在目录名称中使用空格、中划线、中文或无意义缩写。
4. 目录名称必须能够体现职责，不允许使用含糊命名。
5. 不要使用以下临时性、实验性或不可维护的目录名称作为正式结构：
   - temp
   - tmp
   - new
   - old
   - backup
   - final
   - final2
   - misc
   - stuff
   - test1

==============================
二、推荐目录职责划分
==============================

Python 项目原则上应优先使用以下目录命名：

1. src/
   - 放项目核心源码
   - 业务代码、模块代码、正式 Python 包应放在 src/ 下
   - 不应把测试代码、构建产物、发布产物放入 src/

2. tests/
   - 放测试代码
   - 单元测试、集成测试、smoke test 均应放在 tests/ 下
   - 不应把正式业务代码放入 tests/

3. scripts/
   - 放开发辅助脚本、运维脚本、一次性工具脚本、启动脚本
   - 不应把长期维护的核心业务逻辑堆积在 scripts/ 下
   - 若某段逻辑长期存在且被重复调用，应沉淀到 src/ 中

4. rules/
   - 放规则、约束、工程规范、协作说明
   - 若项目采用 rules/ 作为规则目录，则应统一使用 rules/
   - 不要同时长期混用 docs/ 和 rules/ 表达同类规则内容，除非职责明确区分

5. docs/
   - 放面向用户或开发者的说明文档、设计文档、使用文档
   - 若项目使用 rules/ 存放规则，则 docs/ 应仅用于文档，不应重复承担规则目录职责

6. config/ 或 configs/
   - 放配置文件、配置模板、静态配置数据
   - 同一项目中只能选一种命名风格并保持一致
   - 不要同时混用 config/ 和 configs/

7. assets/
   - 放静态资源文件，如图片、图标、模板、示例资源等
   - 不要把运行时生成产物放到 assets/

8. data/
   - 放项目依赖的数据文件、样例数据、小型静态数据
   - 不要默认将大型可下载数据直接提交进仓库
   - 用户生成数据或运行时数据应与 data/ 区分

9. build/
   - 放构建中间产物
   - 只用于构建过程生成的中间文件
   - 不得放置源码、手工维护文件或最终发布内容

10. release/
   - 放最终发布产物
   - 所有面向最终使用者交付的软件包、发布目录、压缩包或可分发结果，统一放在 release/
   - 不使用 dist/ 作为发布目录

11. examples/
   - 放示例代码、最小样例、演示材料
   - 不要把正式测试或正式业务逻辑混放在 examples/

12. notebooks/
   - 放 Jupyter Notebook
   - 不要把正式生产逻辑长期只保存在 notebooks/ 中

13. logs/
   - 放日志文件
   - 不得放置源码、测试代码或发布产物

14. cache/ 
   - 放缓存文件或临时文件
   - 只能选一种风格并保持一致
   - 不得承载正式源码、正式数据或发布内容

==============================
三、源码目录规范
==============================

1. src/ 下应尽量采用清晰的 Python 包结构组织代码。
2. 顶层业务包名称应简洁、稳定、可读，并使用小写下划线风格，例如：
   - src/mario_auto_player/
   - src/image_viewer/
   - src/storage_tools/

3. src/ 下的顶层 Python 包名称应尽量与项目核心语义一致。
4. 不要在 src/ 下创建职责模糊的目录，例如：
   - common
   - utils
   - helper
   - helpers
   - misc
   - temp

5. 如果确实需要公共模块，应优先使用更具体的职责命名，例如：
   - file_utils/
   - config_loader/
   - audio_processing/
   - model_runtime/

6. 除非职责确实无法进一步细分，否则不要默认创建笼统的 utils/、common/、helpers/ 目录。

==============================
四、测试目录规范
==============================

1. 测试统一放在 tests/ 下。
2. tests/ 下可按测试类型或模块进一步划分，例如：
   - tests/unit/
   - tests/integration/
   - tests/smoke/

3. 测试文件名称应清晰表达被测试对象或测试目的。
4. 不要把测试代码散落在项目根目录。
5. 除非项目已有明确历史规范，否则不要把测试代码长期放在 src/ 内。

==============================
五、脚本目录规范
==============================

1. 开发辅助脚本统一放在 scripts/ 下。
2. scripts/ 中的文件名称应体现明确用途，例如：
   - scripts/run_local.py
   - scripts/check_env.py
   - scripts/build_release.py

3. scripts/ 主要用于入口封装、辅助操作、开发工具，不应替代正式模块设计。
4. 若脚本逻辑长期存在、被多个地方复用或逐渐复杂，应沉淀到 src/ 中，再由 scripts/ 调用。

==============================
六、产物目录规范
==============================

1. 构建中间产物统一放在 build/。
2. 最终发布产物统一放在 release/。
3. 不使用 dist/ 作为发布目录。
4. 不得同时使用 build/ 和其他重复语义目录存放中间产物。
5. 不得同时使用 release/ 和 dist/ 表达最终发布目录。
6. 不要把临时文件、缓存文件、构建结果、发布结果散落在项目根目录。
7. release/ 只能承载最终交付内容，不应混入中间构建文件。

==============================
七、命名一致性规则
==============================

1. 同一项目中，同类职责目录必须保持命名一致。
2. 不要同时出现多套含义重复的目录，例如：
   - script/ 和 scripts/
   - test/ 和 tests/
   - doc/ 和 docs/
   - config/ 和 configs/
   - cache/ 和 caches/
   - release/ 和 dist/

3. 一旦项目确定了一种命名风格，后续新增目录必须沿用同一风格。
4. 若历史项目已存在不一致命名，不要无故大规模重命名；应先说明问题，再提出最小化调整方案。

==============================
八、禁止事项
==============================

未经我明确允许，你不得：

1. 创建职责模糊的目录名称
2. 创建临时性命名目录作为长期正式结构
3. 把源码、测试、脚本、构建产物、发布产物混放在同一目录
4. 把构建产物提交到 src/ 或 tests/
5. 把核心业务逻辑长期堆积在 scripts/ 或 notebooks/ 中
6. 使用 dist/ 作为发布目录
7. 随意新增与现有目录重复语义的目录
8. 因局部方便而破坏整体目录一致性

==============================
九、执行规则
==============================

1. 在新建 Python 项目时，应优先遵循本规范建立目录结构。
2. 在调整现有项目目录时，应优先保持职责清晰、命名统一、迁移最小化。
3. 每次新增目录时，必须说明该目录的职责和预期内容。
4. 每次调整目录结构时，必须说明：
   - 为什么要调整
   - 涉及哪些目录
   - 是否影响导入路径、构建流程、测试流程、发布流程
5. 若项目已有 Makefile，则目录调整后必须同步检查 clean、build、test、release 相关逻辑是否需要更新。
