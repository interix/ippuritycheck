你在处理任何 Swift 项目时，必须遵守以下目录创建规范。除非我明确允许，否则不得偏离。

==============================
一、总体原则
==============================

1. Swift 项目的目录结构必须体现职责清晰、命名统一、可维护、可扩展。
2. 源码、测试、资源、构建产物、发布产物必须分离，不得混放。
3. 目录创建时应优先兼容 Swift Package Manager（SPM）的标准习惯。
4. 若项目为 Apple 平台应用（如 macOS/iOS App），也应尽量保持源码、资源、测试、脚本、构建产物的职责清晰。
5. 不得创建临时性、含糊或不可维护的正式目录结构。

==============================
二、命名基本规则
==============================

1. 所有目录名称默认使用英文。
2. 目录命名应简洁、稳定、可读，体现明确职责。
3. 顶层工程目录可根据项目命名使用：
   - PascalCase
   - lower_snake_case
   但同一项目内必须保持一致风格。

4. 不允许使用以下目录名作为正式长期结构：
   - temp
   - tmp
   - new
   - old
   - backup
   - final
   - final2
   - misc
   - stuff
   - test1

5. 不要使用含糊目录名，如：
   - common
   - helper
   - helpers
   - misc
   - utils
   除非职责非常明确且短期无法更精确命名。

==============================
三、推荐顶层目录结构
==============================

Swift 项目原则上应优先使用以下顶层目录：

1. Sources/
   - 放正式源码
   - 所有 Swift 核心模块、业务代码、库代码应优先放在 Sources/ 下

2. Tests/
   - 放测试代码
   - 单元测试、集成测试、UI 测试（如适用）均应放在 Tests/ 下

3. Resources/
   - 放项目运行所需静态资源
   - 如图片、模板、配置模板、示例资源、静态数据等
   - 不要把构建产物放入 Resources/

4. Scripts/
   - 放开发辅助脚本、构建辅助脚本、检查脚本、自动化脚本
   - 不应把正式核心业务逻辑长期堆积在 Scripts/ 下

5. Docs/ 或 rules/
   - Docs/ 用于用户文档、设计说明、开发说明
   - rules/ 用于规则、协作要求、工程规范
   - 若两者同时存在，职责必须清晰区分，不得重复承担同类内容

6. Config/ 或 Configs/
   - 放配置模板、工程配置文件、静态配置数据
   - 同一项目中只能选一种风格并保持一致

7. Assets/
   - 放静态资源、设计资源、图标等
   - 如果项目采用 Apple 平台默认资源管理方式，也可将资源整合到平台工程资源目录中
   - 但职责必须清晰

8. Build/
   - 放构建中间产物
   - 不得放置源码、手工维护文件或最终发布内容

9. Release/
   - 放最终发布产物
   - 所有最终交付给使用者的软件包、压缩包、安装包、发布目录，统一放在 Release/

10. Examples/
   - 放示例项目、最小示例、演示用例
   - 不得把正式源码和正式测试长期混放在 Examples/ 中

11. Logs/
   - 放日志文件

12. Cache/ 或 Tmp/
   - 放缓存文件或临时文件
   - 同一项目中只能选一种风格并保持一致

==============================
四、Sources/ 规范
==============================

1. 所有正式 Swift 源码应优先放在 Sources/ 下。
2. 对于 Swift Package 项目，应遵循 SPM 常见结构：
   - Sources/<ModuleName>/

3. 顶层模块名称应使用清晰、稳定、可读的命名，例如：
   - Sources/MarioPlayer/
   - Sources/ImageViewer/
   - Sources/StorageTools/

4. 不要在 Sources/ 下创建职责模糊的目录，例如：
   - Common
   - Helpers
   - Misc
   - Utils
   除非职责已经足够明确。

5. 应优先使用更具体的模块或职责命名，例如：
   - AudioProcessing/
   - ConfigLoader/
   - FileSystem/
   - UIComponents/
   - AppCore/

6. 如果项目规模较大，可在模块下继续按职责细分，例如：
   - Sources/ImageViewer/App/
   - Sources/ImageViewer/Features/
   - Sources/ImageViewer/Models/
   - Sources/ImageViewer/Services/
   - Sources/ImageViewer/Views/

==============================
五、Tests/ 规范
==============================

1. 所有测试代码统一放在 Tests/ 下。
2. 对于 Swift Package 项目，测试目录应尽量与模块对应，例如：
   - Tests/MarioPlayerTests/
   - Tests/ImageViewerTests/

3. 测试目录名称应体现被测试模块或测试类型。
4. 不得把正式业务代码放入 Tests/。
5. 不要把测试代码散落在项目根目录或 Sources/ 中，除非项目已有明确历史规范。

==============================
六、Resources/ 与 Assets/ 规范
==============================

1. Resources/ 用于项目运行时依赖的静态资源。
2. Assets/ 更适合放设计资源、静态素材、图标原文件等。
3. 如果项目使用 Apple 平台资源体系（如 Asset Catalog），应保持资源入口清晰，不得让资源散落各处。
4. 不得把构建产物、缓存文件、发布文件放入 Resources/ 或 Assets/。

==============================
七、Scripts/ 规范
==============================

1. Scripts/ 用于开发辅助、构建辅助、自动化辅助。
2. Scripts/ 中的脚本名称应体现明确用途，例如：
   - Scripts/build_release.sh
   - Scripts/check_env.sh
   - Scripts/run_local.sh

3. Scripts/ 不应长期承载核心业务逻辑。
4. 若某段逻辑会长期维护、被重复调用或逐渐复杂，应优先沉淀到 Sources/ 中，再由脚本调用。

==============================
八、Build/ 与 Release/ 规范
==============================

1. Build/ 统一用于构建中间产物。
2. Release/ 统一用于最终发布产物。
3. 不得把构建中间产物写入源码目录。
4. 不得把最终发布内容散落在项目根目录。
5. 不得混用多个重复语义目录表达相同职责。
6. Release/ 只能承载最终交付内容，不应混入中间构建文件。

==============================
九、Apple 平台应用项目补充规范
==============================

1. 如果项目是 macOS/iOS/watchOS/tvOS 应用，可根据平台工程结构保留 App 入口目录。
2. 但仍应尽量保证源码、资源、测试、脚本、构建产物分离。
3. 推荐至少具备以下职责划分：
   - App/：应用入口、App 生命周期
   - Features/：功能模块
   - Models/：数据模型
   - Services/：服务层
   - Views/：界面层
   - Resources/：资源文件

4. 不要把所有 Swift 文件长期平铺在单一目录中。
5. 若使用 Xcode 工程组（Groups）与物理目录不同步，应优先说明实际磁盘目录结构，不要只描述 IDE 分组。

==============================
十、命名一致性规则
==============================

1. 同一项目中，同类职责目录必须保持命名一致。
2. 不要同时出现多套重复语义目录，例如：
   - Script/ 和 Scripts/
   - Test/ 和 Tests/
   - Config/ 和 Configs/
   - Build/ 和 Builds/
   - Release/ 和 Releases/

3. 一旦项目确定一种命名风格，后续新增目录必须沿用同一风格。
4. 若历史项目存在不一致命名，不要无故大规模重命名；应先说明问题，再提出最小化调整方案。

==============================
十一、禁止事项
==============================

未经我明确允许，你不得：

1. 创建职责模糊的目录名称
2. 创建临时性命名目录作为长期正式结构
3. 把源码、测试、脚本、构建产物、发布产物混放在同一目录
4. 把构建产物提交到 Sources/ 或 Tests/
5. 把核心业务逻辑长期堆积在 Scripts/ 中
6. 因局部方便而破坏整体目录一致性
7. 只调整 Xcode 工程分组而不说明实际磁盘目录结构

==============================
十二、执行规则
==============================

1. 在新建 Swift 项目时，应优先遵循本规范建立目录结构。
2. 在调整现有 Swift 项目目录时，应优先保持职责清晰、命名统一、迁移最小化。
3. 每次新增目录时，必须说明该目录的职责和预期内容。
4. 每次调整目录结构时，必须说明：
   - 为什么要调整
   - 涉及哪些目录
   - 是否影响构建流程、测试流程、资源加载或发布流程
5. 若项目已有 Makefile，则目录调整后必须同步检查 clean、build、test、release 逻辑是否需要更新。
