====================
UI界面自动化测试要求
====================

本项目属于 macOS 原生图形界面应用，并包含主 App、File Provider Extension、Quick Look Extension 等多个目标。因此，测试体系不能只覆盖纯逻辑层，还必须覆盖图形界面层的自动化验证。凡是用户可见、可交互、对功能正确性或可用性有显著影响的 UI 路径，都必须纳入自动化测试设计范围。

本章节用于约束 Claude 在设计、实现、重构和补测试时，必须同步考虑 UI 自动化测试，而不是只生成业务代码或单元测试代码。

### 1. 总体原则

1. UI 测试必须作为正式测试体系的一部分，而不是可有可无的补充；
2. 对图形界面程序，必须区分：
   - 逻辑测试；
   - UI 自动化测试；
   - 必要时的快照测试；
3. 默认以 Apple 原生测试框架为主：
   - XCTest
   - XCUITest
4. 对纯逻辑模块，可使用 XCTest 或 Swift Testing；
5. 对真实界面交互、菜单、窗口、按钮、表单、设置页、状态提示、升级提示等用户操作路径，必须优先使用 XCUITest；
6. 对视图渲染、重要状态页、复杂布局或稳定视觉输出，可补充快照测试；
7. 不允许把“手动点一点看起来能用”视为 UI 测试完成；
8. 所有关键 UI 交互路径都应尽量具备可重复执行的自动化测试。

### 2. UI测试框架要求

在本项目中，UI 自动化测试默认采用以下分层：

1. XCTest：
   - 用于测试基础逻辑、状态机、数据层、配置层、更新逻辑、版本比较、归档状态映射等；
2. XCUITest：
   - 用于测试窗口、菜单、按钮、输入框、表单、设置页、更新提示、账号配置流程等；
3. Swift Testing：
   - 可用于补充逻辑层和模型层测试，但不替代 XCUITest；
4. Snapshot Testing（如采用）：
   - 用于关键 UI 页面、状态页、设置页或特殊渲染结果的快照校验。

如果引入第三方 UI 测试或快照测试框架，必须说明：
1. 为什么需要它；
2. 它与原生测试框架的边界；
3. 它在工程中的放置方式；
4. 它如何与现有 Makefile、CI、测试门禁配合。

### 3. 必须覆盖的UI测试范围

以下界面和交互流程，原则上都应纳入自动化测试设计：

1. 应用首次启动流程；
2. 主窗口打开与关闭；
3. 账号添加、编辑、删除流程；
4. 多云 provider 选择与配置流程；
5. endpoint / region / bucket 配置界面；
6. 设置页关键选项保存与恢复；
7. 文件浏览主界面中的基础交互；
8. 本地缓存相关提示和操作入口；
9. 解冻/恢复归档对象的操作入口和提示；
10. 自动升级相关界面，包括：
   - 检查更新
   - 发现新版本提示
   - 下载更新进度
   - 重启安装提示
11. 错误提示、失败提示和异常状态页面；
12. 首次安装后的初始化提示；
13. 必要时的菜单栏、偏好设置窗口、关于窗口；
14. 任何对用户关键路径有直接影响的确认框、警告框、迁移提示框。

对于 Finder 深度集成、File Provider Extension、Quick Look Extension 相关功能，若无法全部用标准 XCUITest 直接覆盖，也必须说明：
1. 哪些部分可直接自动化；
2. 哪些部分需要通过项目内部测试入口、mock 环境或专门测试工具辅助验证；
3. 哪些部分只能做半自动或集成级测试；
4. 对这些无法完全 UI 自动化的能力，最小可执行测试策略是什么。

### 4. UI可测试性要求

Claude 在编写或重构 UI 代码时，必须主动考虑自动化测试可行性。所有重要 UI 都应具备良好的可测试性，不允许把界面写成难以定位、难以断言、难以自动化操作的结构。

至少应满足以下要求：

1. 关键按钮、输入框、菜单项、列表项、状态标签应具备稳定可定位的标识；
2. 必须优先使用明确、稳定的 accessibility identifier 或等价机制；
3. 不允许让测试严重依赖易变的显示文案、随机顺序、临时布局层级；
4. 对状态敏感界面，应提供清晰、可断言的状态表达方式；
5. 对加载中、失败、成功、禁用、警告等状态，应有可识别的 UI 标记；
6. 对异步流程，应设计清晰的等待条件，避免测试仅依赖固定 sleep；
7. 对可重复出现的对话框、弹窗、确认框，应统一设计可识别的元素结构；
8. 若某些原生系统弹窗难以自动化，应明确隔离边界，并尽量通过封装降低测试脆弱性。

### 5. UI测试设计原则

所有 UI 自动化测试必须遵守以下原则：

1. 优先测试用户真实操作路径，而不是只测试内部实现细节；
2. 优先覆盖关键主流程，而不是把测试资源平均消耗在边缘路径上；
3. 每个 UI 测试用例应有明确目的，避免写成过长的大杂烩脚本；
4. 一个测试用例只验证少数几个强相关目标；
5. 对相同类型的交互，应尽量抽取公共操作辅助方法；
6. 测试命名必须体现“场景 + 操作 + 预期结果”；
7. 不允许大量堆砌脆弱的点击链而没有断言；
8. 测试应尽量可重复、可并行、可在干净环境下运行；
9. 对耗时较长的 UI 测试，应合理分层，不要所有测试都强制走完整路径；
10. 对环境依赖较强的界面，应尽量使用测试专用配置、mock 数据或假 provider。

### 6. UI测试分层要求

UI测试至少应分为以下层次：

1. 基础可打开性测试
   - 应用能否启动
   - 关键窗口能否显示
   - 基础菜单能否打开

2. 核心路径测试
   - 添加 provider 账号
   - 打开设置
   - 修改配置并保存
   - 触发关键操作并验证结果

3. 状态变化测试
   - 归档对象状态变化提示
   - 自动升级提示状态
   - 错误和异常状态切换
   - 配置变化后的界面反馈

4. 回归测试
   - 过去出现过问题的 UI 场景
   - 与多云 provider 配置相关的历史问题
   - 与缓存、升级、初始化、表单校验相关的历史问题

5. 可选的快照测试
   - 设置页
   - 状态页
   - 错误页
   - 升级提示窗口
   - 重要渲染页面

### 7. Claude在输出代码时的UI测试要求

当 Claude 编写任何 UI 相关代码时，必须同步考虑是否需要新增或修改对应的 UI 自动化测试，并主动输出以下内容：

1. 该 UI 功能对应的测试点；
2. 需要新增的 XCUITest 用例；
3. 若界面结构变化，是否需要更新现有 UI 测试；
4. 若新增可见状态，是否需要新增断言；
5. 若新增设置项、菜单项、弹窗或升级提示，是否需要新增测试入口；
6. 若新增 accessibility identifier，应明确说明其命名与用途；
7. 若某个 UI 功能难以测试，应明确说明原因，并给出改善方案；
8. 不允许只改界面实现，不补充测试影响分析。

### 8. UI测试代码组织要求

测试目录中必须为 UI 自动化测试单独建立清晰目录或 target，不允许与纯逻辑单元测试混杂无序放置。

建议至少包含：

1. UI测试 target；
2. UI测试辅助工具目录；
3. 公共页面操作封装；
4. 测试用 fixture / 假数据；
5. 测试环境配置；
6. 必要时的快照基线目录。

推荐按功能组织测试，而不是按杂乱脚本堆积。可考虑类似结构：

- `Tests/UITests/`
- `Tests/UITests/Flows/`
- `Tests/UITests/Pages/`
- `Tests/UITests/Helpers/`
- `Tests/UITests/Fixtures/`
- `Tests/SnapshotTests/`

### 9. UI测试的环境控制要求

为了让 UI 自动化测试稳定可重复，必须支持测试环境隔离。Claude 在设计工程时应优先考虑：

1. 测试模式启动参数；
2. 测试专用配置文件；
3. 测试专用 provider 数据；
4. 测试专用假账号或 fake backend；
5. 测试时关闭不必要的联网依赖；
6. 测试时避免真实升级、真实删除、真实高风险写操作；
7. 测试时允许注入 mock 服务、固定时间、固定返回值；
8. 必须尽量避免 UI 测试依赖真实云环境才能运行。

如需网络交互，应优先使用：
1. mock feed
2. fake provider
3. 本地测试数据
4. 受控集成测试环境

### 10. UI测试与构建门禁的关系

UI 自动化测试必须纳入整体测试门禁设计，但可按层次控制执行策略：

1. 所有核心逻辑测试必须在 build 前通过；
2. 所有基础 UI 自动化测试应尽量纳入标准测试流程；
3. 耗时较长或环境依赖较强的 UI 测试，可按 smoke / full / integration 分组；
4. 至少应有一组最小 UI smoke tests，作为构建前或发布前的基本门禁；
5. package 或 release 前，应至少执行：
   - 核心单元测试
   - 状态机测试
   - 关键 UI smoke tests
6. 若某些 UI 测试未纳入默认 build 前执行，则必须在文档和 Makefile 中明确说明其分组和触发方式。

### 11. 与Makefile和发布流程的关系

UI 自动化测试必须在 Makefile 或等价测试流程中体现。至少需要考虑：

1. `make test` 中是否包含 UI 测试；
2. 是否区分：
   - 单元测试
   - UI smoke tests
   - 完整 UI tests
3. 是否提供类似下列 target：
   - `make test`
   - `make test-unit`
   - `make test-ui`
   - `make test-ui-smoke`
4. package / release 前需要执行哪些 UI 测试；
5. UI 测试失败时，构建和发布是否中止；
6. UI 测试运行所需环境在文档中必须写清楚。

### 12. UI测试最低执行标准

若当前任务涉及新增、修改或重构图形界面，则最低要求包括：

1. 明确新增或受影响的 UI 测试点；
2. 至少补充对应的 UI 自动化测试骨架；
3. 对关键路径新增断言；
4. 对关键控件补充可定位标识；
5. 若暂时无法自动化，也必须说明原因、风险和后续补测方案。

如果缺少上述内容，则视为该次 UI 功能实现不完整。
