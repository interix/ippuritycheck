====================
计划文件归档与项目内持久化要求
====================

在使用 Claude 进行项目分析、方案设计、实现拆解、重构规划和测试规划时，所生成的 code plan、implementation plan、refactor plan、test plan 等计划性文档，原则上不应仅作为临时会话产物存在于全局目录中。凡是与当前项目直接相关、并对后续开发、测试、维护、评审或追溯具有长期参考价值的计划文件，都必须同步归档到项目目录中，作为项目文件长期保存。

### 1. 适用范围

以下类型的计划文件，原则上都属于需要项目内归档的对象：

1. 与当前项目直接相关的架构设计方案；
2. 与当前项目直接相关的模块实现计划；
3. 跨多个文件或多个模块的重构计划；
4. 涉及数据库、状态机、缓存策略、构建流程、测试策略的专项方案；
5. 与发布、安装、打包、依赖准备、环境初始化相关的实施计划；
6. 对后续代码实现具有明确指导意义的任务拆解计划；
7. 对排障、迁移、兼容性处理、多云 provider 适配等具有持续参考价值的分析计划。

以下内容通常不要求强制归档到项目内，除非用户明确要求保留：

1. 一次性临时推演；
2. 明显属于中间草稿且不会继续引用的短期计划；
3. 与当前项目无直接关系的泛化建议；
4. 没有形成明确执行价值的零散思路记录。

### 2. 基本原则

1. 不允许将具有长期项目价值的计划文件仅保存在全局 `~/.claude/plans/` 目录中；
2. 对当前项目有持续价值的计划文件，必须在项目目录中保留一份可追溯、可版本管理的副本；
3. 项目内归档的计划文件应视为项目文档的一部分，而不是临时缓存；
4. 后续代码实现、测试设计、重构和文档更新，应优先参考项目内归档的计划文件；
5. 若计划内容发生重大调整，必须同步更新项目内归档版本，而不是只更新全局 plans 中的临时版本；
6. 若同一主题存在多个版本，应通过命名或版本字段明确区分，不允许无序覆盖；
7. 归档动作应尽量在计划生成后尽快完成，不应长期拖延。

### 3. 项目内保存位置

计划文件应保存到项目内固定目录，不允许随意散落。推荐目录如下：

- `docs/plans/`
- `.claude/project-plans/`

默认优先推荐：

- 对需要参与团队协作、代码评审、版本管理的正式计划，放在 `docs/plans/`
- 对偏 Claude 工作流、Agent 工作记录但仍需项目持久化的计划，可放在 `.claude/project-plans/`

如无特殊说明，建议统一使用：

- `docs/plans/`

### 4. 命名规范

项目内归档的计划文件必须采用统一命名规则，文件名建议包含以下信息：

1. 日期；
2. 任务主题；
3. 可选版本号；
4. 可选计划类型。

推荐命名格式示例：

- `YYYY-MM-DD-task-topic-plan.md`
- `YYYY-MM-DD-task-topic-v2.md`
- `YYYY-MM-DD-provider-adapter-refactor-plan.md`
- `YYYY-MM-DD-file-provider-cache-design.md`

命名要求如下：

1. 文件名应稳定、可读、可搜索；
2. 不允许仅使用“plan.md”“new-plan.md”“temp.md”这类无语义名称；
3. 如果同一主题有多个版本，应显式加入 `v2`、`v3` 等版本标记；
4. 若计划属于特定类型，可在文件名中增加类型标识，例如：
   - `architecture-plan`
   - `implementation-plan`
   - `refactor-plan`
   - `test-plan`

### 5. 文件内容要求

归档到项目中的计划文件应尽量整理成可长期阅读和复用的文档，而不是保留大量会话化表述。归档版本至少应包含：

1. 计划标题；
2. 背景与目标；
3. 适用范围；
4. 主要设计或实施步骤；
5. 关键约束与假设；
6. 风险、待验证项与未决问题；
7. 与代码、模块、目录或测试的关联关系；
8. 如有必要，补充版本信息和更新时间。

如果全局 plans 中的原始内容过于口语化、零散或中间态，应在归档到项目目录时进行适度整理，使其更适合作为项目长期文档保存。

### 6. 执行要求

Claude 在进行项目相关工作时，应遵循以下规则：

1. 当生成新的项目级计划时，必须判断该计划是否具有长期参考价值；
2. 如果具有长期参考价值，则必须将其保存到项目固定目录中；
3. 在开始实现、重构或测试前，应优先检查项目目录中是否已有相关计划文件；
4. 如果已有相关计划文件，应在其基础上延续，而不是重复生成多个相互割裂的计划；
5. 如果计划内容发生重大变化，应更新项目中的归档版本；
6. 如果生成了新的替代性计划，应明确说明其与旧计划的关系；
7. 不允许长期依赖全局 `~/.claude/plans/` 作为唯一计划来源。

### 7. 与开发流程的关系

项目内归档的计划文件应当与以下流程形成关联：

1. 架构设计；
2. 模块实现；
3. 数据库变更；
4. 状态机设计；
5. 多云 provider 适配；
6. File Provider 行为设计；
7. 缓存与同步机制；
8. 测试设计；
9. Makefile 与构建发布流程；
10. 安装、注册与系统集成流程。

当上述流程涉及较大改动时，优先要求先更新或新增对应计划文件，再进入代码实现阶段。

### 8. 与版本管理的关系

1. 项目内归档的计划文件应默认纳入项目版本管理；
2. 计划文件应作为可审阅、可追溯的工程文档；
3. 对重大设计调整，应保留必要的版本演进痕迹；
4. 不建议无控制地覆盖历史计划；
5. 如同一主题持续演进，可采用版本号或更新记录方式维护。

### 9. Prompt 约束与自动化实现的边界

1. 本章节定义的是 Claude 在项目工作中的行为规范；
2. Claude 应主动遵守“项目相关计划必须归档到项目目录”的要求；
3. 但仅靠 prompt 约束并不能替代自动化机制；
4. 若项目后续提供 hook、agent、脚本或命令来自动复制、重命名、索引和归档计划文件，应优先与本章节配合使用；
5. 在存在自动化机制时，本章节仍然作为高层规则有效；
6. 在缺乏自动化机制时，Claude 也应尽最大努力手动完成归档动作和归档说明。

### 10. 最低执行标准

若当前项目尚未建立完整的计划归档自动化机制，则最低要求如下：

1. 与项目直接相关的重要计划文件必须在项目目录中保留一份副本；
2. 保存位置应固定；
3. 命名应统一；
4. 后续实现应以项目内副本为准；
5. 若计划发生重大变化，项目内副本必须同步更新。

如未满足上述最低标准，则视为项目文档归档流程不完整。
