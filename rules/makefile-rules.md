你在处理任何软件项目时，必须严格遵守以下 Makefile 工程规范。除非我明确解除限制，否则这些规则始终有效。

==============================
一、总体原则
==============================

1. 每个项目都必须在项目根目录提供 Makefile。
2. Makefile 必须作为项目工程操作的统一入口。
3. 与依赖准备、依赖检查、清理、构建、测试、发布相关的核心流程，必须统一体现在 Makefile 中，不允许长期依赖零散的手工命令作为正式入口。
4. 如果项目内部确实依赖其他脚本、工具链或配置文件，Makefile 仍必须作为统一入口对其进行封装、调用或组织。
5. 不允许只修改脚本、文档或临时命令而不更新 Makefile。
6. 依赖准备、环境准备、构建、测试、发布应尽量自动化、可重复执行、可审计、可维护。
7. Makefile 应体现项目的完整工程生命周期，而不仅仅是编译命令集合。

==============================
二、必备 target
==============================

Makefile 至少必须包含以下 target：

1. help
   - 用于列出主要 target 及其用途
   - 应作为默认 target，或至少作为推荐入口

2. deps
   - 用于准备本项目所需的全部依赖
   - 包括但不限于：
     - 虚拟环境准备
     - 依赖库安装
     - 模型文件准备
     - 外部资源文件下载
     - 项目运行前必须准备的基础环境
   - deps 应作为依赖准备的统一入口

3. check-deps
   - 用于检查本项目依赖是否已经准备齐全
   - 检查内容包括但不限于：
     - 虚拟环境是否存在
     - 依赖库是否安装
     - 模型文件是否存在
     - 关键外部资源是否可用
     - 依赖版本是否满足约束
   - check-deps 应尽量为只读检查，不应默认执行破坏性修改

4. clean
   - 用于清理可安全删除的临时文件、缓存文件、构建产物和明确无关文件

5. build
   - 用于将源码、资源文件和必要配置转换为可运行程序、可执行文件或标准构建产物

6. test
   - 用于执行项目测试、最小验证流程或 smoke test

7. release
   - 用于将构建产物整理为最终可交付的发布包或发布目录
   - 最终使用者应能够基于 release 产物直接获得和使用软件

如果项目规模较小，某些 target 暂时无法完整实现，也必须保留对应 target，并在注释或输出中明确说明当前限制和后续建议。

==============================
三、deps 规范
==============================

1. deps 是项目依赖准备的统一入口。
2. deps 负责准备本项目开发、构建、测试、运行或发布所必需的全部依赖。
3. deps 可准备的内容包括但不限于：
   - Python 虚拟环境
   - 第三方依赖库
   - 模型文件
   - 数据文件
   - 外部二进制工具
   - 项目运行所需基础资源
4. deps 的执行过程应尽量自动化，不应依赖手工零散步骤。
5. 如果项目使用 Python，则依赖准备逻辑应尽量统一收敛到单一脚本，例如：
   - scripts/prepare_deps.py
6. 不允许为依赖准备长期维护多个职责重复、来源分散、难以追踪的零碎脚本。
7. 若依赖准备需要辅助函数或模块，可在源码中拆分实现，但 Makefile 层面的统一入口必须收敛为单一 prepare_deps 主脚本或单一统一执行链路。
8. deps 应尽量具备幂等性，重复执行不应造成不可预期副作用。
9. 若 deps 包含下载、安装、覆盖或写入操作，必须在输出中明确说明将要执行的主要动作。
10. 若 deps 会下载大文件、模型文件或大型资源，应明确提示体积、目标目录和用途。
11. 每当项目新增依赖时，必须同步补齐 deps 逻辑，不允许新增依赖后仍依赖手工准备。
12. 不允许出现“README 要求手工执行额外依赖命令，但 Makefile 的 deps 未体现”的情况。

==============================
四、check-deps 规范
==============================

1. check-deps 是项目依赖状态检查的统一入口。
2. check-deps 的职责是确认本项目依赖是否已经满足，不应默认承担完整依赖安装职责。
3. check-deps 应尽量优先做只读检查，并明确输出：
   - 哪些依赖已满足
   - 哪些依赖缺失
   - 哪些版本不符合要求
   - 建议下一步动作
4. 如果项目依赖准备复杂，check-deps 应尽量做到可读、明确、可快速定位问题。
5. check-deps 可与 deps 配合使用，但两者职责必须区分：
   - deps 负责准备
   - check-deps 负责检查
6. 若项目存在关键依赖缺失，build、test、release 在执行前应优先依赖 check-deps，或至少显式提示依赖状态问题。

==============================
五、依赖脚本统一入口规范
==============================

1. 如果项目使用脚本实现依赖准备逻辑，则依赖准备应尽量统一由单一主脚本负责，例如：
   - scripts/prepare_deps.py
2. 不允许为依赖准备长期散落维护多个零碎脚本，例如：
   - install_x.py
   - download_model.py
   - fix_env.py
   - prepare_runtime.py
   若这些逻辑确有必要存在，也应由单一主脚本统一调度，而不是让使用者分别执行。
3. Makefile 中的 deps target 应显式调用这个统一入口。
4. check-deps 也应有统一入口，例如：
   - scripts/check_deps.py
   或由同一主脚本支持检查模式，但其职责必须清晰可读。
5. 不允许依赖准备逻辑长期隐藏在未说明的脚本中。

==============================
六、clean 规范
==============================

1. clean 只能删除可再生成内容，包括但不限于：
   - 构建产物
   - 缓存文件
   - 临时文件
   - 中间产物
   - 明确无关的自动生成文件

2. 未经我明确批准，不得通过 clean 删除以下内容：
   - 源码文件
   - 配置文件
   - 用户数据
   - 测试数据
   - 模型文件
   - 手工维护文档
   - 其他不可恢复内容

3. 在执行 clean 之前，必须先明确说明：
   - 将要删除的文件或目录
   - 删除原因
   - 影响范围
   - 是否存在高风险内容

4. 只有在得到我的明确批准后，才可以执行实际清理操作。

5. 如果 clean 涉及大目录、大体积文件、历史产物、缓存仓库、用户生成内容或 release 内容，必须视为高风险操作，单独提醒，不得默认执行。

==============================
七、build 规范
==============================

1. build 必须实现从源码到可运行程序、可执行文件或标准构建产物的转换。
2. build 过程应尽量自动化，不应依赖手工复制、手工编辑或手工整理。
3. build 过程应尽量可重复执行，同样输入条件下应得到一致或等价的输出结果。
4. build 输出目录必须明确，例如：
   - build/
   - out/
   - bin/
5. 不允许让构建产物无规则地散落在项目目录中。
6. 如果 build 依赖前置环境、工具链或资源，必须在 Makefile 中体现依赖关系，或在输出中明确说明。
7. build 应尽量依赖 check-deps，必要时提示先执行 make deps。

==============================
八、test 规范
==============================

1. test 必须作为项目标准流程的一部分存在于 Makefile 中。
2. test 的职责是验证当前源码或构建结果的基本正确性。
3. 如果项目已有完整测试体系，test 应调用正式测试流程。
4. 如果项目暂时没有完整测试体系，test 也至少应提供以下之一：
   - 最小 smoke test
   - 基础启动验证
   - 占位实现并明确说明缺失项
5. build、release 相关逻辑调整后，如测试逻辑需要同步调整，也必须更新 Makefile。
6. test 应尽量依赖 check-deps，确保依赖状态明确。

==============================
九、release 规范
==============================

1. release 必须实现从构建产物到最终发布包、发布目录或可交付软件形态的转换。
2. release 面向最终使用者，而不是仅面向开发者。
3. release 产物应尽量做到开箱可用，或至少具备明确的运行说明。
4. release 应尽量明确以下信息：
   - 软件名称
   - 版本信息
   - 目标平台信息
   - 必要资源文件
   - 运行方式或使用说明（如适用）

5. release 输出目录必须明确，例如：
   - release/
   - artifacts/

6. release 不应只是简单重复 build；它应体现“面向交付”的整理过程，例如：
   - 收集可执行文件
   - 收集必要资源
   - 整理目录结构
   - 生成发布包
   - 输出最终可分发结果

7. release 应尽量建立在 build 成功、test 通过、依赖状态明确的前提之上。

==============================
十、统一入口与一致性规范
==============================

1. deps、check-deps、clean、build、test、release 的正式流程必须统一体现在 Makefile 中。
2. 每次对依赖准备、依赖检查、清理、构建、测试、发布逻辑做出调整时，都必须同步更新 Makefile。
3. 不允许出现以下情况：
   - README 中命令已变，但 Makefile 未更新
   - 依赖准备脚本已变，但 Makefile 未更新
   - 构建脚本已变，但 Makefile 未更新
   - 发布流程已变，但 Makefile 未更新
   - 清理逻辑已变，但 Makefile 未更新

4. 如果实际逻辑由脚本完成，Makefile 也必须显式调用这些脚本并体现调用关系。

==============================
十一、默认 target 与安全性规范
==============================

1. Makefile 的默认 target 应为 help，或其他安全、不具破坏性的 target。
2. 不得将 clean 设为默认 target。
3. 不得将 release 设为默认 target。
4. 不得将任何高风险操作设为默认 target。
5. deps 可作为推荐入口之一，但若其包含大体积下载或明显副作用，应通过 help 明确提示。

==============================
十二、幂等性与可维护性规范
==============================

1. deps、check-deps、clean、build、test、release 应尽量具备幂等性。
2. 重复执行同一 target 时，不应产生不可预期副作用。
3. Makefile 中的 target 名称应清晰、稳定、可理解，不要使用含糊命名。
4. Makefile 应尽量提供必要注释，便于后续维护。
5. 若存在平台差异、依赖差异或特殊限制，必须明确说明。
6. 依赖准备脚本应尽量具备自动跟进能力；当项目依赖变化时，应优先补齐统一脚本，而不是新增手工步骤。

==============================
十三、修改 Makefile 时的说明义务
==============================

每次新增或修改 Makefile 后，你必须同步说明以下内容：

1. 新增或修改了哪些 target
2. 每个 target 的职责
3. 各 target 的执行前提条件
4. 输出目录和输出内容
5. 是否影响现有工作流
6. 是否存在兼容性风险
7. 是否需要我额外确认某些操作
8. 若新增依赖，是否已同步补齐 deps 和 check-deps 逻辑
9. 若依赖准备脚本有变动，统一入口是否仍然保持清晰

==============================
十四、执行规则
==============================

1. 在你新增项目或调整工程结构时，应优先检查 Makefile 是否存在及是否完整。
2. 如果 Makefile 缺失，应优先补齐 Makefile，而不是只提供零散命令。
3. 如果 Makefile 不完整，应优先将 deps、check-deps、clean、build、test、release 补齐或整理到统一入口。
4. 在建议工程流程时，应优先建议使用：
   - make help
   - make deps
   - make check-deps
   - make build
   - make test
   - make release
   而不是零散命令。
5. 在执行 clean 之前，必须先征求我的明确批准。
6. 在发现项目新增依赖时，必须主动检查 Makefile 和统一依赖准备脚本是否需要同步更新。
